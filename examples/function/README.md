# p03 职能拆分解耦 — 实验方案

## 实验目的

验证将业务拆分为标准职能（课程/咨询/产品）的粒度是否合理，职能间组合与依赖关系是否可管理。

## 背景

日志 2026-05-31："最困扰我的问题是怎么分拆成不同的业务……拆分成课程、咨询、产品这些职能就好。"

## 实验假设

1. 业务可以正交分解为有限个标准职能
2. 职能间通过明确定义的输入输出接口组合，不会产生循环依赖
3. 同一职能在不同业务流程中可复用

## 实验设计

### 阶段 1：职能定义

定义以下基础职能的 DSL 描述：

| 职能 ID | 名称 | 输入 | 输出 | 前置依赖 |
|---------|------|------|------|---------|
| F01 | 课程研发 | 课程大纲 | 课件/教材/习题 | 无 |
| F02 | 咨询服务 | 客户需求 | 咨询方案/报告 | 无 |
| F03 | 产品开发 | 产品需求 | MVP/文档 | F05 |
| F04 | 市场推广 | 产品/课程 | 获客线索 | F01/F02/F03 |
| F05 | 技术支持 | 技术需求 | 平台/工具 | 无 |

### 阶段 2：业务流程组合

用已定义的职能描述以下业务：

- **企业培训方案** = F01 + F04（客户→课程→推广）
- **数字化转型咨询** = F02 + F05（客户→诊断→技术方案）
- **SaaS 产品交付** = F03 + F05 + F04（需求→开发→推广）

### 阶段 3：依赖分析

- 检测循环依赖
- 检测缺失依赖
- 验证组合规则是否完备

### 阶段 4：正交性检测

量化检测各职能间的职责重叠度：

```
正交性检查规则（自动执行）：
  1. 输入集交集 / 并集 < 30%  → 通过
  2. 输出集无完全包含关系    → 通过
  3. 职责描述无 50% 以上关键词重叠 → 通过
  4. 同时满足三者 → 正交

示例：F01 课程研发（输入：课程大纲）vs F02 咨询服务（输入：客户需求）
  - 输入交集/并集：0%（正交）
  - 职责关键词重叠："需求"出现在两个描述中，需人工判断上下文
```

## 评价标准

| 指标 | 合格线 | 优秀线 |
|------|--------|--------|
| 职能正交性（无重叠职责） | 无两个职能职责交集 > 30% | 无两个职能职责交集 > 10% |
| 组合规则可覆盖实际业务 | 3 种 | 5+ 种 |
| 循环依赖检测 | 100% 检出 | 100% 检出 |
| 新增职能不影响已有组合 | 新增后现有组合通过率 ≥ 90% | 新增后现有组合 100% 通过 |

## 原型实现

职能编辑器 + 依赖图可视化，包含：

- 职能列表（增删改）
- 职能-输入-输出编辑器
- 依赖图渲染（DAG）
- 组合规则配置面板
- 循环依赖告警

## 实验记录

- 每次新增/修改职能后的依赖图变化
- 组合规则的验证结果（通过/失败）
- 发现的重叠或遗漏的职能
